# Next.js + Firestore 上市計畫書

本計畫書目標是協助目前的社團活動平台，以最低風險、可控成本方式，從既有 PHP + MySQL 架構逐步轉向 Next.js + Firebase（Firestore），並完成對外上線。

## 1. 費用問題先回答（重點版）

結論：可以先低成本甚至接近 0 成本起步，但正式上市後通常會有費用。

### 1.1 可能免費的部分

- Next.js 部署到 Vercel Hobby：可免費起步（有使用量限制）。
- Firebase Spark（免費層）：Firestore、Auth、Storage 都有免費額度。
- 開發與測試環境：在免費額度內通常可維持 0 元。

### 1.2 何時會開始花錢

- 流量上升，超過 Firestore 讀寫與儲存免費額度。
- 圖片海報上傳增加，Storage 與網路傳輸費用上升。
- 需要更多進階功能（例如高頻通知、排程、進階監控）。
- 團隊進入商用協作，Vercel 可能需升級 Pro。

### 1.3 預估區間（保守估算）

- MVP / 小規模驗證（校內少量使用者）：每月約 0 到 500 TWD。
- 校內穩定使用（中等流量）：每月約 500 到 3,000 TWD。
- 跨校或持續成長（高流量）：每月約 3,000 TWD 以上。

注意：實際費用高度依賴 Firestore 查詢設計與圖片流量，若查詢未優化，成本會放大。

## 2. 專案目標

- 在 8 到 10 週內完成可上市版本。
- 保留既有核心流程：社團瀏覽、活動報名、評價、QA、幹部管理、管理端。
- 建立可持續維運的部署與監控流程。

## 3. 目標架構

- 前端與 BFF：Next.js（App Router）。
- 身分驗證：Firebase Auth。
- 主資料：Firestore。
- 檔案：Firebase Storage（活動海報、社團 logo）。
- 部署：Vercel（Web）+ Firebase（資料與儲存）。
- 監控：Vercel Analytics + Firebase Console。

## 4. 遷移策略（建議）

採用「漸進式遷移」而非一次重寫。

1. 第 1 階段先做新前端骨架與登入。
2. 第 2 階段以 API 相容方式逐步替換既有頁面。
3. 第 3 階段切換資料來源到 Firestore，保留回退方案。
4. 第 4 階段完成全量切換後下線舊入口。

## 5. 工作分期與里程碑

## 5.1 Phase A（第 1-2 週）：啟動與設計

- 定義 Firestore Collections 與索引策略。
- 完成角色權限模型（學生、幹部、管理員）。
- 建立 CI/CD、環境變數與分環境部署。

交付物：
- 資料模型文件。
- 權限規則草案。
- Next.js 專案骨架。

## 5.2 Phase B（第 3-4 週）：核心流程 MVP

- 完成登入註冊與使用者狀態。
- 完成社團列表、社團詳情、活動列表、活動詳情。
- 完成圖片上傳流程。

交付物：
- 可操作的前台 MVP。
- 主要頁面 Lighthouse 基線報告。

## 5.3 Phase C（第 5-6 週）：互動功能

- 完成活動報名流程。
- 完成評價與 QA 流程（含禁詞規則）。
- 完成通知中心與個人資料頁。

交付物：
- 互動功能驗收報告。
- 基本壓力測試結果。

## 5.4 Phase D（第 7-8 週）：管理端與上線準備

- 完成幹部儀表板（社團資料、活動管理、海報更新）。
- 完成管理員頁面（公告、審核、帳號轉讓流程）。
- 建立監控、告警、備援與回退 SOP。

交付物：
- 上線檢核清單。
- 正式環境部署紀錄。

## 6. Firestore 資料設計建議（摘要）

主要 collections：

- users
- clubs
- events
- registrations
- reviews
- qa_questions
- qa_replies
- notifications
- tags

設計原則：

- 優先以畫面查詢路徑設計資料，不先追求關聯正規化。
- 高頻列表頁採冗餘欄位，減少多次查詢。
- 所有需要排序/篩選的查詢先規劃複合索引。
- 審核與權限相關欄位不可只做前端限制，必須落在安全規則。

## 7. 風險與對策

- 風險：Firestore 成本失控。
  - 對策：建立查詢配額監控、限制無限滾動、減少重複讀取。

- 風險：複雜查詢難以維護。
  - 對策：先定義查詢白名單，必要時以 Next.js API Route 做聚合。

- 風險：資料遷移造成資料不一致。
  - 對策：雙寫期間加校驗報表，提供回滾腳本。

- 風險：上線後圖片流量暴增。
  - 對策：縮圖、WebP、快取策略與圖片尺寸限制。

## 8. 驗收指標（KPI）

- 首頁 LCP 小於 2.5 秒。
- 核心流程成功率（登入、報名、評論）高於 99%。
- 嚴重等級 bug（P1）為 0。
- 上線首月可用性高於 99.5%。

## 9. 預算與人力建議

最小人力配置（建議）：

- 1 位前端（Next.js）
- 1 位全端/後端（Firebase + 資料模型）
- 1 位測試與上線支援（可兼任）

若單人開發，可把 8 到 10 週拉長到 12 到 16 週，降低風險。

## 10. 上線檢核清單

- 已完成正式環境 env 設定。
- 已完成安全規則審核與測試。
- 已建立備份與回退流程。
- 已完成管理端權限驗證。
- 已完成壓力測試與成本預估更新。

## 11. 建議決策

- 若你目標是快速上市驗證：採 Next.js + Firebase 可行。
- 若你預期未來查詢關聯會很複雜：可評估 Next.js + PostgreSQL（如 Supabase）作為主資料庫。
- 最保守方案：先做前端 Next.js 化，資料層維持 MySQL，第二階段再切 Firestore。
