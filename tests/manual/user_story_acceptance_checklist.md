## User Story 4.1

- 管理員登入後可進入管理頁並新增社團。
- 必填欄位留空時應出現驗證錯誤。
- 停用或隱藏的社團不應出現在前台列表。
- 置頂公告應在首頁以明顯樣式顯示。
- 社團主要幹部帳號數量應受規則限制。

## User Story 2.1

- 幹部登入後只能管理自己的社團。
- 更新社團時，必填欄位不可留空。
- 欄位留空時應顯示紅字提示。
- 儲存成功後，社團詳情應立即更新。
- 編輯時應可看到預覽區塊。

## User Story 2.2

- 建立活動時，名稱、日期時間、地點與內容為必填。
- 任一必填欄位空白時不能發布。
- 發布後應在社團詳情的近期活動中可見。
- 過期活動不應出現在前台清單。

## User Story 1.1

- 社團列表應可依分類篩選。
- 社團列表應支援關鍵字搜尋。
- 無結果時應顯示空狀態文案。

## User Story 1.5

- 幹部更新社團資料後，詳情頁應顯示最後更新時間。
- 發布新活動後，最後更新時間應同步改變。

## User Story 1.3

- 學生追蹤社團後，按鈕樣式應改變。
- 再次點擊應取消追蹤。
- 個人頁應顯示追蹤社團的未過期活動。
- 未追蹤社團時應顯示空狀態與前往探索按鈕。

## 驗收方式

1. 以管理員、幹部與學生帳號分別登入。
2. 依序操作每個使用情境。
3. 確認畫面、驗證與資料更新符合預期。
4. 將失敗項目記錄到測試報告。
